📋 Spec Gate - 의도를 구조화하라

모호한 요청을 구조화된 스펙으로 변환합니다.
"코드 생성 전에 스펙부터 확정" - 이것이 VibeOps의 핵심입니다.

💬 기존: 모호한 요청

개발자
"로그인 기능 만들어줘"
AI (기존)
네! Google OAuth로 소셜 로그인을 구현하겠습니다...
❌ OAuth 금지 정책 위반 (AI가 모름)

✨ VibeOps: 스펙 먼저

개발자
"로그인 기능 만들어줘"
⚡ VibeOps Spec Gate
스펙을 확인해 주세요:

feature: 사용자 로그인
auth_method: JWT (자체 인증)
constraints: OAuth 금지 [C001]
hash_algorithm: bcrypt

이대로 진행할까요? [Y/n/edit]

📄 자동 생성되는 스펙 파일

# VibeOps가 자동 생성한 스펙 spec: id: SPEC-001 feature: 사용자 로그인 created: 2026-02-03 status: confirmed requirements: - 이메일/비밀번호 기반 인증 - JWT 토큰 발급 (자체 인증) - 비밀번호 bcrypt 해싱 constraints: - ref: C001 # OAuth 금지 original_request: "로그인 기능 만들어줘"

🔍 Verify Gate - 충돌을 사전 감지

새 요청이 기존 결정/정책과 충돌하는지 코드 생성 "전"에 감지합니다.

⚠️ 충돌 감지 시나리오

개발자
"소셜 로그인도 추가해줘"
⚡ VibeOps Verify Gate
⚠️ 정책 충돌 감지!

요청: 소셜 로그인 추가
충돌: [C001] OAuth/소셜 로그인 사용 금지
이유: 보안 정책상 외부 IdP 연동 불가

[1] 정책 유지 → 자체 인증으로 구현
[2] 정책 수정 → 예외 추가 후 진행
[3] 취소

❌ 없이: 재작업 발생

✅ Validate Gate - 생성 후 자동 검증

AI가 코드를 생성한 "후" 자동으로 품질을 검증합니다.

🔍 검증 항목

🛡️

정책 검사

금지 키워드/패턴이 코드에 포함되었는지 검사

🔒

보안 검사

하드코딩된 비밀, SQL 인젝션 등 취약점 감지

🎨

일관성 검사

기존 코드베이스 스타일/패턴과 일치 여부

📊 검증 결과 예시

═══════════════════════════════════════════════════════════════ VibeOps Validate Gate - 검증 결과 ═══════════════════════════════════════════════════════════════ 📁 파일: src/auth/login.py ✅ 정책 검사: PASSED • OAuth 관련 코드 없음 • 외부 CDN 참조 없음 ✅ 보안 검사: PASSED • 하드코딩된 비밀 없음 • bcrypt 해싱 사용 확인 ✅ 일관성 검사: PASSED • 네이밍 컨벤션 준수 ═══════════════════════════════════════════════════════════════ 총점: 100/100 - 커밋 가능 ✓ ═══════════════════════════════════════════════════════════════

📝 Auto Document - 자동 문서화

모든 스펙과 결정이 자동으로 문서화됩니다. "왜 이렇게 만들었지?"에 답합니다.

📁 자동 생성되는 구조

.vibe/ ├── specs/ # 기능별 스펙 │ ├── login.yaml │ └── payment.yaml ├── constraints/ # 정책/제약 │ └── policies.yaml ├── decisions/ # 기술 결정 │ └── decisions.yaml └── context.yaml # 프로젝트 컨텍스트

🔍 vibe why 명령어

$ vibe why src/auth/login.py 📁 파일: src/auth/login.py 📋 스펙: SPEC-001 (로그인 기능) 🎯 원본 요청: "로그인 기능 만들어줘" 📌 적용된 결정: • D001: JWT 자체 인증 (OAuth 대신) • D002: bcrypt 해싱 ⛔ 적용된 제약: • C001: OAuth 사용 금지
