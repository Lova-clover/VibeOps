🎯 문제 정의

바이브 코딩에는 품질 관리 프로세스가 없습니다

핵심 문제

전통적 개발에는 요구사항 → 설계 → 개발 → 테스트 → 리뷰의 프로세스가 있습니다.
반면 바이브 코딩은 "만들어줘" → 코드 → 커밋으로 끝나는 경우가 많습니다.

❌ 현재 바이브 코딩

요청"로그인 기능 만들어줘"
AI 생성OAuth 코드 생성 (금지 정책을 모름)
커밋검토 없이 바로 커밋
나중에"왜 이렇게 만들었지?" - 맥락 소실

✅ VibeOps 적용

1. Spec요청을 구조화된 스펙으로 변환
2. Verify"OAuth 금지 정책과 충돌" 경고
3. Generate확정된 스펙 기반 코드 생성
4-5자동 검증 + 자동 문서화

📊 왜 프로세스가 필요한가?

항목 전통적 개발 현재 바이브 코딩 VibeOps 적용
스펙 문서화필수 (기획서)없음자동 생성
코드 리뷰동료 리뷰없거나 형식적자동 검증
정책 준수가이드라인AI가 모름자동 주입
의도 추적ADR 문서대화에서 소멸자동 기록

📋 Spec Gate

모호한 요청을 구조화된 스펙으로 변환합니다

💬 기존 방식

개발자
"로그인 기능 만들어줘"
AI (기존)
네! Google OAuth로 소셜 로그인을 구현하겠습니다...
⚠️ OAuth 금지 정책 위반

✨ VibeOps 적용

개발자
"로그인 기능 만들어줘"
⚡ VibeOps Spec Gate
스펙을 확인해 주세요:

feature: 사용자 로그인
auth: JWT (자체 인증)
constraints: OAuth 금지 [C001]

이대로 진행할까요? [Y/n/edit]

📄 자동 생성되는 스펙 파일

# VibeOps가 자동 생성한 스펙 spec: id: SPEC-001 feature: 사용자 로그인 requirements: - 이메일/비밀번호 기반 인증 - JWT 토큰 발급 - 비밀번호 bcrypt 해싱 constraints: - ref: C001 # OAuth 사용 금지

🔍 Verify Gate

기존 정책과의 충돌을 코드 생성 전에 감지합니다

⚠️ 충돌 감지 예시

개발자
"소셜 로그인도 추가해줘"
⚡ VibeOps Verify Gate
⚠️ 정책 충돌 감지

요청: 소셜 로그인 추가
충돌: [C001] OAuth/소셜 로그인 사용 금지
사유: 보안 정책상 외부 IdP 연동 불가

[1] 자체 인증으로 구현 [2] 정책 예외 추가 [3] 취소

❌ Verify Gate 없을 때

1분AI가 OAuth 코드 생성
30분개발자가 통합 테스트
2시간코드 리뷰에서 정책 위반 발견
4시간+코드 재작업

✅ Verify Gate 적용 시

1초요청 분석
2초정책 충돌 감지
5초대안 제시
완료올바른 방향으로 진행

✅ Validate Gate

AI가 생성한 코드를 자동으로 검증합니다

🛡️

정책 검사

금지된 키워드/패턴이 코드에 포함되었는지 검사합니다

🔒

보안 검사

하드코딩된 비밀, SQL 인젝션 등 기본 취약점을 감지합니다

🎨

일관성 검사

기존 코드베이스의 스타일/패턴과 일치하는지 확인합니다

📝 Auto Document

모든 스펙과 결정이 자동으로 문서화됩니다

📁 자동 생성되는 구조

.vibe/ ├── specs/ # 기능별 스펙 │ ├── login.yaml │ └── payment.yaml ├── constraints/ # 정책/제약 │ └── policies.yaml ├── decisions/ # 기술 결정 │ └── decisions.yaml └── context.yaml # 프로젝트 컨텍스트

🔍 vibe why 명령어

$ vibe why src/auth/login.py 📁 파일: src/auth/login.py 📋 스펙: SPEC-001 (로그인 기능) 🎯 원본 요청: "로그인 기능 만들어줘" 📌 적용된 결정: • D001: JWT 자체 인증 (OAuth 대신) • D002: bcrypt 해싱 ⛔ 적용된 제약: • C001: OAuth 사용 금지
